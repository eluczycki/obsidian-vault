# Depradar — v0.4 Plan

## Goal

Complete system isolation: app/server writes its own status, OpenClaw monitors from the outside and reports problems. Clear separation of concerns — app stays self-contained, monitoring is Claw's job.

## Architecture

```
Coolify Server (depradar)          Azure VM (OpenClaw)
┌─────────────────────────┐        ┌─────────────────────────┐
│ /api/status             │        │ Cron job (every 5 min)   │
│   → system status        │  ←curl │   → fetch /api/status    │
│   → app runtime status   │        │   → evaluate health      │
│   → script/error logs    │        │   → post to Discord      │
│                           │        │     #1501238142265593946 │
│ scripts post to /api/... │        │                         │
│   → daily-repo-check     │        │                         │
│   → db-health            │        │                         │
│   → etc.                 │        │                         │
└─────────────────────────┘        └─────────────────────────┘
```

## Components

### 1. `GET /api/status` — Single endpoint, always JSON

Returns current system + app state. Structured for easy polling.

**Response shape:**
```json
{
  "timestamp": "2026-05-09T20:00:00Z",
  "system": {
    "uptimeSeconds": 86400,
    "memoryUsedMb": 128,
    "memoryLimitMb": 512
  },
  "app": {
    "version": "0.3.0",
    "env": "staging",
    "dbConnected": true,
    "dbLatencyMs": 12
  },
  "scripts": {
    "lastRepoCheck": {
      "ranAt": "2026-05-09T08:00:00Z",
      "status": "success",
      "checked": 250,
      "updated": 3,
      "failed": 1,
      "error": null
    },
    "lastSeed": {
      "ranAt": "2026-05-09T16:00:00Z",
      "status": "success",
      "seeded": 250,
      "errors": 0
    }
  },
  "alerts": [
    {
      "level": "warning",
      "code": "DB_LATENCY_HIGH",
      "message": "DB latency exceeded 500ms",
      "since": "2026-05-09T19:45:00Z"
    }
  ]
}
```

### 2. Scripts on the server write status to a shared log or POST directly

**Option A — scripts write to a local JSON log file, status endpoint reads it:**
```
/app/data/script-status.json  ← written by scripts
/api/status reads this file + adds system/app metrics
```
Simple, no external deps. Scripts just do `fs.writeFile`.

**Option B — scripts POST their results to `/api/status/report`:**
```
POST /api/status/report { script, status, data }
→ appends to script log array
→ status endpoint reads from that log
```
Cleaner API, scripts self-report. More work.

**Recommendation: Option A** — simpler, scripts don't need to know about the API contract, just write a JSON file. Status endpoint reads and enriches.

### 3. OpenClaw cron job monitors `/api/status`

Simple Node.js script (no AI needed for monitoring):
- `curl https://stg.dprdr.co/api/status`
- Evaluate: `alerts.length > 0`, `dbConnected === false`, `memoryUsedMb/memoryLimitMb > 0.9`
- If problem: format Discord message, send to `#1501238142265593946`
- If all clear: do nothing (no noise when healthy)

**No LLM involvement** — just HTTP fetch + simple conditional logic + Discord API call.

### 4. Discord alert format

```
⚠️ Depradar Status — YYYY-MM-DD HH:MM UTC

🔴 [CRITICAL] Database disconnected

🟡 [WARNING] DB latency 620ms (threshold: 500ms)
```

## Self-contained scripts on the server

Any script that runs on the Coolify server should be in `scripts/` and just write to `data/script-status.json`. No AI, no OpenClaw involvement.

Example: `scripts/daily-repo-check.ts`:
```typescript
// Runs on server cron, writes its own status
const result = await fetch(...);
// do work
await fs.writeFile(
  path.join(process.cwd(), 'data/script-status.json'),
  JSON.stringify({
    script: 'repo-check',
    ranAt: new Date().toISOString(),
    status: result.failed > 0 ? 'warning' : 'success',
    ...result
  })
);
```

## v0.4 Additions (post-initial plan)

### GC Sweeper — ghost package cleanup
Packages that accumulate CVEs but have `last_commit = null` and no `github_repo` after N days are ghosts — likely missing repo links or bad data ingested via search bypass. GC logic:
- `last_refreshed > 7 days ago` AND `github_repo IS NULL` AND `cve_count > 0` → flag for review
- After N consecutive GC cycles (configurable): auto-delete or mark `status = 'ghost'`
- Results written to `script-status.json` for the monitoring cron to pick up
- Admin endpoint: `POST /api/admin/purge-ghosts` for manual cleanup

### ON CONFLICT explicit constraint naming
All upserts use `ON CONFLICT (column)` which infers the constraint. Should name explicitly:
```sql
ON CONFLICT ON CONSTRAINT packages_npm_name_key DO UPDATE SET ...
```
Prevents edge cases where unique index names diverge from column inference.

### Coolify deploy trigger fix
Currently: push to `staging` branch triggers a Coolify deploy webhook, but the build can fail silently or hang. The API has no reliable deploy/rebuild endpoint.
- Investigate: is there a working `POST /deployments` or similar endpoint?
- Fallback: add a "deploy" button in the Coolify UI or a manual webhook script
- Goal: iteration speed — ability to redeploy on demand, not just on push

### Wire-up production system
- Deploy to `dprdr.co` (not just staging)
- Test full stack: `/api/packages`, `/api/packages/[name]`, `/api/packages/lookup`, `/api/packages/refresh`, `/api/reported`
- Test Report flow end-to-end (submit → view in `/api/reported` → adjust package → clear report)
- Merge `staging` → `master` and confirm prod deployment via Coolify

### Direct seed: npm → DB (no intermediate JSON)
**Current:** `POST /api/seed` reads from `public/data/packages.json` → 250 package hardcoded list. `fetchPackages.ts` prebuild step was removed because it hit GitHub rate limits at build time.

**v0.4 path:** Runtime seed that fetches from npm registry directly:
- `GET /api/packages/lookup?q=<name>` already does DB-first, then live fetch from npm/GitHub/OSV
- `POST /api/seed` should call `fetchPackageData()` for each package in `packages.ts` (the curated 250 list) rather than reading from static JSON
- `scripts/fetchPackages.ts` becomes a runtime script (can run via `POST /api/seed`) not a build-time prebuild
- Benefit: fresh data on every seed call, no stale JSON, no build-time rate limiting

**Implementation:**
- `POST /api/seed` calls `fetchPackageData()` for each entry in `src/lib/packages.ts`
- Remove `public/data/packages.json` dependency
- `fetchPackages.ts` stays in `scripts/` as a reference/debug tool
- Seed becomes idempotent and can be re-run anytime

### Verification / checkup scripts
Before v0.4 ships, we need a quick sanity check that the system is healthy.

**`scripts/verify-db.ts`** — runs on the server (or via SSH):
1. DB connection + latency ping
2. Package count (expect ~250)
3. Ghost packages: `github_repo IS NULL AND cve_count > 0 AND last_refreshed < 7 days ago`
4. Data freshness: oldest `last_refreshed`, any null `last_commit` where there shouldn't be
5. `reported_packages` count (for report feature sanity check)

**`scripts/seed-db.ts`** — direct seed script (runs standalone, not via `POST /api/seed`):
- Reads from `src/lib/packages.ts` (the curated 250 list)
- Computes risk score live (npm + GitHub + OSV fetch)
- Writes directly to DB via pg client
- Can be run from the Coolify server via `docker exec` or SSH
- Exit codes: 0 = success, 1 = error

**Note:** `scripts/seedDb.ts` and `scripts/check_payload.js` and `scripts/cleanup_test_pkgs.js` are leftover debug scripts — clean up before prod.

### Report feature — future expansions (v0.4+)

Future ideas:
